1. WHAT WE ARE BUILDING (HACKATHON SCOPE)
================================================================================

PermitOS is a multi-agent permitting command center for real estate developers.

A developer uploads:
  - Project address
  - Project type (e.g. 50-unit residential)
  - Plans / PDFs or structured project brief (sq ft, stories, units, parking)

Five specialized agents analyze the project in parallel, coordinate through Band,
merge findings, resolve conflicts, assemble a permit package, and produce an
audit-ready trail. A human approves before anything is marked "ready to file."

WHAT WE SHIP FOR THE HACKATHON
------------------------------
[+] Web UI: upload project + live agent activity feed (Band-backed)
[+] 5 Band-connected agents with distinct roles and (ideally) different frameworks
[+] One polished demo jurisdiction: Austin, TX (city + Travis County + key state triggers)
[+] Structured outputs: compliance report, conflict list, permit checklist, fee estimate
[+] Human approval gate with audit log
[+] 3–5 minute demo video + GitHub repo

WHAT WE DO NOT NEED FOR THE HACKATHON
-------------------------------------
[-] Live integration with every city permit portal API
[-] All 50 US states
[-] Auto-submit to government systems
[-] CAD/plan parsing perfection — structured brief + optional PDF text extract is enough

2. THE 5 AGENTS (FINAL ROSTER)
================================================================================

We merged 8 agents into 5 by combining related domains. Each agent is a separate
Band participant with its own system prompt, tools, and (preferably) framework.

--------------------------------------------------------------------------------
AGENT 1: CONDUCTOR (Orchestrator + Conflict Resolver + Human Gate)
--------------------------------------------------------------------------------
Role:       Project intake, task dispatch, merge results, resolve conflicts, escalate
Framework:  LangGraph or PydanticAI (recommended — good for structured orchestration)
Model:      Strong reasoning model (Claude / GPT-4 class)

Responsibilities:
  - Receives project from UI, creates/opens Band chatroom for the permit case
  - Parses intake into structured ProjectBrief JSON
  - @mentions specialist agents with scoped tasks (not the whole blob)
  - Waits for specialist agents to post structured findings
  - Detects conflicts between agents (e.g. zoning OK but fire egress fails)
  - Proposes resolution paths or design modifications
  - Computes overall readiness score (Ready / Needs Changes / Blocked)
  - Escalates to human when: critical conflict, missing data, or final approval
  - Writes final PermitCaseSummary for the dashboard

Outputs (structured JSON):
  - case_id, status, readiness_score
  - conflicts[]: { agents, issue, severity, suggested_fix }
  - human_actions_required[]
  - audit_events[]

Band tools used:
  - thenvoi_create_chatroom (new case)
  - thenvoi_add_participant (invite specialists)
  - thenvoi_send_message (@mention agents)
  - thenvoi_send_event (progress: planning, waiting, merging, escalated)
  - thenvoi_get_participants / thenvoi_lookup_peers

--------------------------------------------------------------------------------
AGENT 2: JURISDICTION & ZONING AGENT
--------------------------------------------------------------------------------
Role:       Which rules apply + land use / zoning compliance
Framework:  CrewAI or Anthropic adapter (different from Conductor if possible)
Model:      Same or different vendor than Conductor (cross-model = judge bonus)

Responsibilities:
  - Resolve address → city, county, state, special districts
  - Load jurisdiction rule pack from knowledge base (Austin MVP)
  - Check: zoning district, permitted use, setbacks, FAR, height, lot coverage,
    parking ratios, density limits
  - Flag if project type is allowed by-right vs requires variance/CUP
  - Cite specific code sections in every finding

Inputs:     address, project type, lot size, unit count, stories
Outputs:     JurisdictionReport JSON

  jurisdictions: [{ name, type, codes_applicable[] }]
  zoning: { district, permitted_use, by_right: bool }
  checks: [{ rule, status: pass|fail|warn, citation, detail }]
  blockers: []
  data_gaps: []

Tools:
  - lookup_jurisdiction(address)        → mock DB or geocoding + rules index
  - get_zoning_rules(city, district)    → RAG over Austin code snippets
  - calculate_setbacks(project, lot)      → deterministic helper

--------------------------------------------------------------------------------
AGENT 3: BUILDING & SAFETY AGENT
--------------------------------------------------------------------------------
Role:       Building code, fire, egress, accessibility pre-screen
Framework:  LangGraph or OpenAI adapter
Model:      Different from Agent 2 if possible

Responsibilities:
  - Pre-screen against IBC/IRC (residential), local amendments
  - Fire: egress paths, sprinkler triggers by occupancy/size, fire lane access
  - Accessibility: ADA / FHA accessible unit %, route requirements
  - Structural red flags from brief (stories, construction type) — not full PE stamp
  - Every finding tagged pass/fail/warn + code citation

Inputs:     ProjectBrief + unit mix, stories, sq ft, parking structure
Outputs:     BuildingSafetyReport JSON (same schema pattern as Agent 2)

Tools:
  - get_building_code_rules(city, project_type)
  - check_egress_requirements(units, stories, corridors)
  - check_accessibility_requirements(unit_count, building_type)

--------------------------------------------------------------------------------
AGENT 4: SITE, ENVIRONMENTAL & UTILITIES AGENT
--------------------------------------------------------------------------------
Role:       Environmental triggers + site infrastructure / utilities
Framework:  PydanticAI or Gemini adapter (third framework = differentiation)
Model:      Third vendor if budget allows

Responsibilities:
  - Environmental: wetlands, floodplain (FEMA zone from address), endangered species
    triggers, stormwater, noise — CEQA/NEPA screening level (not full EIS)
  - Utilities: water/sewer capacity flags, electrical service size, gas,
    storm drain, traffic impact screening
  - Identifies which separate permits/approvals are needed beyond building permit

Inputs:     address, site acreage, unit count, impervious surface estimate
Outputs:     SiteEnvironmentalReport JSON

Tools:
  - lookup_flood_zone(address)            → FEMA mock/API
  - get_environmental_triggers(city, site)
  - get_utility_requirements(project_size, city)

--------------------------------------------------------------------------------
AGENT 5: PERMIT PACKAGER & TRACKER AGENT
--------------------------------------------------------------------------------
Role:       Assemble submission package + track status + draft RFI responses
Framework:  Claude SDK or Codex adapter
Model:      Strong at document generation

Responsibilities:
  - Consumes all specialist reports + Conductor's merged state
  - Generates permit checklist: every form, document, fee, agency, timeline
  - Calculates fee estimates from jurisdiction fee schedule (MVP: Austin table)
  - Produces filing order (what to submit first, dependencies)
  - Simulates post-submission: status states (Submitted → In Review → RFI → Approved)
  - Drafts RFI response when city asks for clarification (demo scenario)

Inputs:     Merged PermitCaseSummary + all specialist reports
Outputs:     PermitPackage JSON

  permits_required: [{ agency, permit_name, form_id, fee, timeline_days, deps[] }]
  documents_required: [{ name, source_agent, status }]
  total_fees_estimate: number
  estimated_timeline_days: number
  filing_sequence: []
  rfi_draft: optional (for demo act 2)

Tools:
  - get_permit_templates(city, project_type)
  - get_fee_schedule(city)
  - generate_checklist(reports[])
  - draft_rfi_response(rfi_text, case_context)

3. HOW BAND IS USED (NOT A THIN WRAPPER)
================================================================================

Band is the ACTIVE coordination layer. The UI talks to Conductor; Conductor and
specialists coordinate in Band chatrooms. State changes happen via Band messages
and events — not just a log dump at the end.

ARCHITECTURE
------------

  [Developer UI]
        |
        v
  [API / Backend]  ---- creates case, stores reports
        |
        v
  [Conductor Agent]  <======= Band Chatroom (per permit case) =======>
        |                    |         |         |         |
        @mentions            |         |         |         |
        v                    v         v         v         v
   [Zoning]            [Building]  [Site/Env]  [Packager]
   Agent               Agent       Agent       Agent

BAND SDK INTEGRATION CHECKLIST (JUDGES CARE ABOUT THIS)
-------------------------------------------------------
[1] Each of the 5 agents registered as separate Band remote agents
[2] Each uses official Band Agent API / SDK adapter
[3] Agents discover each other via thenvoi_lookup_peers or room participants
[4] Task handoffs via thenvoi_send_message with @mentions
[5] Progress via thenvoi_send_event (thoughts, errors, task progress)
[6] Conductor creates room per case via thenvoi_create_chatroom
[7] Structured payloads in messages (JSON blocks, not just prose)
[8] Human added to room for approval step via thenvoi_add_participant

MESSAGE PROTOCOL (KEEP AGENTS ALIGNED)
--------------------------------------
Every agent message includes a JSON block:

  {
    "type": "finding" | "complete" | "question" | "conflict",
    "case_id": "uuid",
    "agent": "jurisdiction" | "building" | "site" | "packager" | "conductor",
    "payload": { ... report or partial ... },
    "citations": [{ "source": "Austin LDC 25-2-491", "url": "..." }]
  }

Conductor only merges on type=complete messages from specialists.

HUMAN-IN-THE-LOOP (TRACK 3 REQUIREMENT)
---------------------------------------
  - Conductor posts PermitPackage + Compliance Summary to room
  - @mentions human user
  - Human replies APPROVE or REQUEST_CHANGES in Band (or via UI that posts to Band)
  - Packager updates status; Audit log records human decision + timestamp
  - No "ready to file" without human APPROVE

AUDIT TRAIL
-----------
  Store locally + surface in UI:
  - Every Band message ID, agent ID, timestamp
  - Every rule checked, pass/fail, citation
  - Human approval record
  - Hash of final package (demo: SHA-256 of JSON) — "provable permit decision"

6. END-TO-END FLOW (SYSTEM BEHAVIOR)
================================================================================

STEP 1 — INTAKE
  Developer submits form in UI
  API creates case_id, stores ProjectBrief
  Conductor agent starts, creates Band room "permit-case-{id}"
  Conductor adds 4 specialist agents to room
  Conductor posts task message with scoped brief to each agent (@mention)

STEP 2 — PARALLEL ANALYSIS (Band)
  Agents 2, 3, 4 run in parallel (Packager waits)
  Each posts thenvoi_send_event("analyzing...")
  Each posts type=complete with structured report JSON
  Typical time: 30-90 seconds per agent in demo

STEP 3 — MERGE & CONFLICT (Conductor)
  Conductor ingests 3 reports
  Runs conflict rules:
    - zoning fail + anything else = BLOCKED
    - building fail + site pass = NEEDS CHANGES
    - conflicting citations → conflict object
  Posts merged summary to room

STEP 4 — PACKAGE (Agent 5)
  Conductor @mentions Packager with merged context
  Packager generates PermitPackage + checklist + fees
  Posts type=complete

STEP 5 — HUMAN GATE
  Conductor posts executive summary + blockers + fees + timeline
  @mentions human
  UI shows approval card; human clicks Approve (posts to Band)
  Audit hash generated

STEP 6 — DEMO ACT 2 (OPTIONAL WOW)
  Simulate RFI from city: "Clarify fire access route on Block B"
  Packager drafts response using case context
  Conductor shows before/after timeline improvement

11. RISKS & MITIGATIONS
================================================================================

Risk                               Mitigation
---------------------------------  ------------------------------------------
Band SDK learning curve            Day 1: one message round-trip before features
Agents return prose not JSON       Strict output schema in prompts; parse/retry
Too slow for live demo             Pre-warm agents; cache Austin RAG; shorten brief
Wrong legal advice                 Label "pre-screen only, not legal advice"; cite sources
One agent framework only           Assign different adapters per agent early
Team blocked on Austin data        Ship with 15 good rules > 100 bad rules
Submission day Band outage         Record demo video Day 6; submit early June 19 AM
